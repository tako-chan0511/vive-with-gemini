# 1. 表紙：絶対に落ちない動的LoRA基盤（全体像）

![IAGI1.完全閉域網で絶対に落ちない動的LoRA基盤](/images/LLM/IAGI1.完全閉域網で絶対に落ちない動的LoRA基盤.jpg)

## 図の要旨

* **「要塞」風の多層構造**で、推論基盤を守る設計思想を提示しています。
* “落ちない”を **思想ではなく、仕組みで保証**する（後続ページの多段防御・試験で証明する）。

## 詳細解説（何が新しいのか）

この資料のキモは「強いGPU」や「でかいモデル」ではなく、**落ちない設計（Failure Containment）** です。

* 落ちる原因は主に3つ

  1. **OOM（GPU/CPUメモリ枯渇）**
  2. **ハング（応答停止・詰まり）**
  3. **運用事故（LoRA差し替え、設定ミス、切り戻し不能）**
* これらを **Kubernetes/OpenShiftのレイヤ**で抑え込むのが方針です。

# 2. ミッション：究極のSovereign AI基盤の構築

![IAGI2.ミッション：究極のSovereignAI基盤の構築](/images/LLM/IAGI2.ミッション：究極のSovereignAI基盤の構築.jpg)

## 図の要旨

* **完全閉域網**で外部ネットワークから物理・論理的に遮断。
* **tsuzumi** をセキュアに運用。
* **限られたGPU資源を複数部門（テナント）で共有**しつつ、再起動・クラッシュを許さない「無停止稼働」を目指す。

## 基本用語

* **Sovereign AI**：自組織の管理下で、データ・モデル・運用を完結させる考え方（主権AI）。
* **テナント**：部門/プロジェクトなどの利用者単位。共用基盤でも“論理分離”が前提。

## Tips（実務）

* ここで最初に決めるべきは「精度」ではなく **SLO**（落とさない、復旧時間、切り戻し時間、許容遅延など）です。
* “無停止”を言うなら、**異常系の仕様（HTTP 400/429/503、タイムアウト、再試行）** まで合意しておくと強いです。

# 3. 制約を越える「プライベートAIキッチン」（完全自給自足）

![IAGI3.製薬御w超える「プライベートAIキッチン」](/images/LLM/IAGI3.製薬御w超える「プライベートAIキッチン」.jpg)

## 図の要旨

外部（Docker Hub / AWS S3 / Hugging Face）へアクセスできない閉域網で、
内部に次を用意して完結させます。

* **Internal Registry**：コンテナ倉庫（イメージを保管）
* **MinIO Storage**：LoRA/データ保管（S3互換）
* **PVC**：巨大Base Model保管（永続ボリューム）

## 詳細解説（なぜこの3点セットが必須か）

閉域網の典型事故は「取りに行けない」です。

* 依存物（モデル、トークナイザ、イメージ、証明書）が外にあり、起動時に詰まる
  → **“最初から外に行かない構成”** が必要です。

## 図外だが必須の実務ポイント

* **内部レジストリへのイメージ搬入**（oc-mirror / skopeo等）
* **MinIOの証明書信頼（Trusted CA bundle）**
* **MinIOアクセス権限（Secret / ServiceAccount）**
* **PVCのReadOnly運用**（誤更新防止）

# 4. なぜ「動的LoRA」が必須なのか？（Pattern比較）

![IAGI4.なぜ「動的LoRA」が必須なのか？](/images/LLM/IAGI4.なぜ「動的LoRA」が必須なのか？.jpg)

## 図の要旨

* **Pattern A（一体型）**：モデルごとにGPU占有 → GPUの深刻な浪費、切替で再起動
* **Pattern B（静的LoRA）**：YAMLにハードコード → 切替で再起動、GPU浪費
* **Pattern C（動的LoRA）**：Baseモデル共有 + 動的ロード → **ゼロダウンタイム** & GPU最適化

## ここでの結論（運用の勝ち筋）

企業では部門/用途が増えるので、モデルを増やすとコストも運用も爆発します。
→ **「Baseは固定」「差分を動的に差し替える」** が唯一スケールする。

## よくある誤解

* 「LoRAは軽いから静的でも良い」
  → 増えるのはLoRA数です。静的だと **追加のたびに運用作業（再起動/停止）が増える**のが問題。

# 5. ハイブリッド・ロード・アーキテクチャ（PVC × MinIO × vLLM）

![IAGI5.ハイブリッドロードアーキテクチャ](/images/LLM/IAGI5.ハイブリッドロードアーキテクチャ.jpg)

## 図の要旨

* **巨大Base Model（tsuzumi）**：PVC `/mnt/models` に固定マウント（ReadOnly推奨）
* **軽量LoRAアダプタ**：MinIO（S3）に保管
* vLLMが **PagedAttention** を活かし、LoRAを **動的注入（Hot-swapping）** する

## 用語

* **Hot-swapping**：サービスを止めずに、必要な差分だけ差し替える動き
* **PagedAttention**：KVキャッシュ等を効率的に扱い、VRAM効率を上げる設計

## Tips（実務）

* LoRA格納は最初から階層を決める（例）：
  `s3://lora/<tenant>/<usecase>/<version>/`
* Baseは**一度ロードして維持**し、LoRAのみを切替えるのが価値です。

# 6. 「落ちない」を担保する多段防御機構（OOM/ハングを封じる）

![IAGI6.「落ちない」を担保する多段防御機構](/images/LLM/IAGI6.「落ちない」を担保する多段防御機構.jpg)

## 図の要旨（守るべき敵）

LLMサービング最大の敵：

* **OOM（メモリ枯渇）**
* **ハングアップ（応答停止）**

これを **インフラレイヤ**で封じ込めます。

## 図の設定（意味）

* `limits.memory`
  → Podに割り当てられるRAM上限。越えるとOOMKill対象（“落とす”のではなく“局所化”する）
* `gpu-memory-utilization=0.9`
  → vLLM側のVRAM使用率上限。余白を確保し、突発負荷での即死を減らす
* `livenessProbe`
  → プロセスハング検知。異常ならPod再起動（自己修復）

## 追加でよく使う設定（図外の鉄板）

* `readinessProbe`：起動直後の未準備状態でトラフィックを受けない
* `startupProbe`：初回ロードが長いLLMの起動を誤判定しない
* `resources.requests`：ノイズネイバー対策（最低保証）

  * 10%カナリアで異常検知 → **瞬時切戻し**

# 8. Phase 1：E2E検証とゼロダウンタイム（異常系で落ちない）

![IAGI8.Phase1：E2E検証とゼロダウンタイム](/images/LLM/IAGI8.Phase1：E2E検証とゼロダウンタイム.jpg)

## 図の要旨

* 正常系：`hr-v1` を **再起動なしで即時ロード**して推論成功
* 異常系：存在しないLoRA（例：`dummy-lora`、JSON欠損/typo）を指定
  → **HTTP 400で拒否**、PodはGreen（Running）維持

## ここが“落ちない”の核心

* 「アプリ側のミス」や「LoRA配置ミス」は必ず起きます
  → その時に **プロセスが落ちず、エラー応答で返す** のが強い。

## よく使う確認コマンド（図外）

* 疎通：`curl -i`（ステータス確認）
* ログ：`oc logs <pod> --tail=200`
* 直近イベント：`oc get events --sort-by=.lastTimestamp`

# 9. Phase 2：キャッシュ動態とパフォーマンス（Cache Miss/Hit）

![IAGI9.Phase2：キャッシュ動態とパフォーマンス](/images/LLM/IAGI9.Phase2：キャッシュ動態とパフォーマンス.jpg)

## 図の要旨

* **Cache Miss（初回）**：MinIOからの取得・VRAM展開で遅延（TTFT↑）
* **Cache Hit（2回目以降）**：高スループット（TPS↑）
* `--max-loras` の範囲なら **LRUキャッシュ**で切替オーバーヘッドが実質ゼロに近づく

## 用語

* **TTFT**：最初のトークンが返るまでの時間（体感の遅さ）
* **TPS**：トークン生成速度（長文の速さ）
* **LRU**：最近使ったものを残し、使われないものから追い出すキャッシュ戦略

## Tips（実務）

* ダッシュボード（Grafana等）で、最低限この4つを見る：

  * TTFT
  * TPS
  * エラー率
  * VRAM使用率
* “初回だけ遅い”は設計通りの場合があるので、**2回目以降の指標**もセットで説明する。

# 10. Phase 3：VRAM枯渇と限界突破（落とすのはPod、落とさないのはサービス）

![IAGI10.Phase3：VRAM枯渇と限界突破](/images/LLM/IAGI10.Phase3：VRAM枯渇と限界突破.jpg)

## 図の要旨

* 超長文 → KVキャッシュ肥大
* 巨大LoRA同時ロード → VRAM/RAM圧迫
* 防波堤：`--max-model-len`、`limits.memory`、Node隔離設計

## 重要な設計思想

* **失敗をゼロにするのではなく、失敗を局所化する**

  * コンテナがOOMKilledされても、全体が落ちなければOK
  * “安全な死に方”を先に設計する

## 代表パラメータ（意味）

* `--max-model-len`：最大コンテキスト長を制限（KVキャッシュ爆発を抑える）
* `limits.memory`：RAM上限（超過は強制終了＝被害の拡大防止）

# 11. Phase 4：オートスケールと自己修復（Knative × Probe）

![IAGI11.Phase4：オートスケールと自己修復](/images/LLM/IAGI11.Phase4：オートスケールと自己修復.jpg)

## 図の要旨

* **Traffic Spike** に対して、Knativeで `N Pods` までスケール
* **livenessProbe** でハング検知 → Kill → 自動復旧（Self-Healing）
* `minReplicas=1` により **コールドスタート回避**（ロードが重いLLMで重要）

## 用語

* **Auto-Scaling**：負荷に応じてPod数を増減
* **Self-Healing**：異常を検知して自動で復旧する仕組み
* **Knative**：KServe等の背後でスケーリング制御を担うことが多い

## Tips（実務）

* “落ちない”要件があるなら、`minReplicas` と `readinessProbe` の設計が重要です。
* スケールアウトは万能ではないので、**GPU資源の上限**（maxReplicasの意味）を事前に合意する。

# 12. Phase 5：カナリアリリースと瞬間切戻し（Kong/アプリで制御）

![IAGI12.Phase5：カナリアリリースと瞬間切り戻し](/images/LLM/IAGI12.Phase5：カナリアリリースと瞬間切り戻し.jpg)

## 図の要旨

* v1に90%、v2（Canary）に10%など、段階的に流す
* 異常（ハルシネーション増・品質劣化）を検知したら **UNDO（瞬間切戻し）**
* LRUキャッシュにより切戻し後も応答性能を維持しやすい

## 重要ポイント（運用）

* カナリアで見るべき指標：

  * 品質（自動評価/人手評価）
  * TTFT/TPSの劣化
  * エラー率増加
* **切戻しが“瞬時”でないと無停止にならない**（ここが設計要件）

# 13. 開発者体験（DX）：OpenAI互換APIで“差し替えを透明化”

![IAGI13.開発者体験（DX）：OpenAI語幹APIの採用](/images/LLM/IAGI13.開発者体験（DX）：OpenAI語幹APIの採用.jpg)

* アプリ開発者は `/v1/chat/completions` 等の **OpenAI互換API** を叩く

```bash
curl -X POST /v1/chat/completions \
  -H 'Content-Type: application/json' \
```

* `-i`：レスポンスヘッダ含め表示（HTTPステータス確認）

## 付録：閉域網で「通信が詰まる」時の切り分け（参考）

> 推論基盤のテストでは、**APIが通らない＝モデルが悪い**ではなく、
> プロキシ・証明書・DNSが原因のことが多いです。
> 詳細は「閉域網 curl 完全攻略ガイド」を参照してください（別資料）。

* `curl -v` でどこで止まったかを証拠化（DNS / TCP / TLS / HTTP）
* `--cacert` で社内ルート証明書を指定（`-k` は恒久策にしない）
* `--resolve` で一時的に名前解決を固定して検証
